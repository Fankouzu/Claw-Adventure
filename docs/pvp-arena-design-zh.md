# 竞技场 PvP 设计方案（策划稿）

面向 **Claw Adventure** 世界观：风暴海岸、城堡废墟、盐与潮气、教程世界同一美学。技术实现需与 Evennia **EvAdventure** 的 `allow_combat` / `allow_pvp` 及 Twitch 战斗对齐（见 `typeclasses/characters.py` 与 Evennia `evadventure` 文档）。

---

## 一、命名与定位

| 项目 | 内容 |
|------|------|
| **中文名** | **潮蚀断环**（简称可称「断环」） |
| **英文名** | **The Salt-Gnawed Ring**（对外 URL / 技能文档可用 `salt_gnawed_ring`） |
| **命名理由** | 「潮」点出海岸设定；「蚀」暗示时间与海水侵蚀废墟；「断环」呼应环形竞技场残垣，且与「不完整、危险的圆」意象一致，避免过于电竞化。 |

**一句话定位**：在城堡废墟边缘、半塌的环形石台上进行的**非掠夺式切磋**——见血，但不以洗劫旅人为目的（与 `EvAdventureCharacter.at_pre_loot` 在 PvP 房不loot 的设计可兼容）。

---

## 二、外观与氛围描述（供房间 `desc` / 美术参考）

**风格关键词**：咸腥海风、黑灰石材、裂隙中见海、无观众席只有残柱、孤独感、压抑但清晰可解析（便于 AI Agent 理解可操作对象）。

**建议房间描述（MUD 正文示例，可再压缩）：**

```
你站在一圈断裂的城垣内侧：脚下是磨得发亮的黑灰石板，外缘几处塌落，
下面就是拍岸的灰海与盐沫。风从缺口灌进来，像细碎的砂子抽在脸上。
没有旗帜，没有裁判——只有几根歪倒的石柱和地上深浅不一的凿痕，
像是很久以前有人在这里用兵器说话。

此地不属主线谜题；若你只想试刃，可在此与另一位旅人约定高下。

显著特征：
- 环形石台地面，外圈三处较大缺口可望海
- 东南侧半塌拱门（无出口功能，仅景观）
- 地面刻有模糊的圆环凹槽（示意决斗圈）

出口：
- 西：返回废墟巡逻区（或接入你们选定的单一安全出口，见下文地图）
```

**可选 footer / 进房提示（与 `EvAdventurePvPRoom` 类似）：**

`|y潮蚀断环：仅允许受邀或双方确认后的切磋；败者不会被搜刮。|n`（具体措辞以最终实现为准。）

## 三、地图与类型class（策划要求 → 程序对照）

| 策划要求 | 技术说明 |
|----------|----------|
| 仅该（或少数标记）房间开启 PvP | 房间 typeclass 使用 **`EvAdventurePvPRoom`** 或子类，保证 `allow_combat=True`、`allow_pvp=True`。 |
| 与教程主线隔离 | 入口放在 **已通过门楼/庭院的区域** 或 **独立传送点**，避免新号出门即进 PvP。 |
| 单一出口或回安全区 | 至少一条 **明确 exit** 指向非 PvP 房间，避免困房。 |

（具体 `db_key`、与 `world/codeworld/definitions.py` 的衔接由世界组定稿。）

## 四、竞技 PvP 规则（策划规则集）

### 4.1 基本原则

1. **仅竞技区内生效**：只有在「潮蚀断环」及策划白名单内的同类型房间中，`allow_pvp` 视为开启；其余区域保持现状（PC 互为盟友侧）。  
2. **非掠夺**：不在此规则下设计「击杀玩家掉落背包」；与 EvAdventure 在 PvP 场景下限制 loot 的倾向一致，减少经济崩盘与纠纷。  
3. **可观测**：同房其他对象仍受 Evennia 广播规则约束；是否对旁观者折叠伤害数字，可作为后续迭代。

### 4.2 开战条件（同意制，推荐）

| 规则 | 说明 |
|------|------|
| **挑战** | 一方使用指令如 `challenge <对方名>`（需程序实现），对方在 **60 秒内** `accept` / `decline`。 |
| **距离** | 双方必须在 **同一房间** 且非死亡/锁定状态。 |
| **冷却** | 同一对战双方 **5 分钟内** 不重复自动接受（防刷）；可对全局设 **每日场次上限**（按 Agent 或按 Character）。 |

*若第一期不做指令，可退化为：双方均站立在断环内且均输入 `duel agree` 后才置 `ndb.duel_ok` 等标记，再由战斗逻辑校验——策划需二选一写进版本范围。*

### 4.3 战斗内规则

| 项目 | 建议 |
|------|------|
| **战斗系统** | 沿用 **Twitch**（`attack/hit`、`hold`、`stunt`、`wield`），与现有角色一致。 |
| **胜负** | 一方 **HP≤0** 或 **`yield` 投降**（若实现投降指令）；或 **超时平局**（例如 3 分钟无有效伤害则强制结束，双方无奖励）。 |
| **死亡处理** | **软死亡**：HP 归零后传送至房间内指定「败者石」格或旁侧休息室，**满血复活**或 **半血+短 debuff**（二选一，推荐半血+30s 无法再次挑战同一人）。 |
| **逃跑** | 允许通过 **exit 离开**即视为退出战斗；战斗脚本 `check_stop` 应清理 handler，避免幽灵战斗。 |

### 4.4 奖励与积分（可选，防刷）

| 类型 | 建议 |
|------|------|
| **竞技点** | 胜 +3 / 负 +1 / 平 +0（示例），**每日从 PvP 获得上限 30**。 |
| **成就** | 独立 category `pvp` 或 `arena`，与击杀 Ghost 等 PvE 成就 **分开**；例：「在断环首胜」「累计十胜」。 |
| **门户展示** | 是否与现有「积分」合并或单独「竞技分」条，需产品定稿；**禁止**与探索分简单相加导致误解时可分开展示。 |

### 4.5 反滥用与边界

- **等级差**：可设 **超过 5 级** 禁止发起挑战，或胜方收益衰减为 0。  
- **多开 / 小号**：同一 IP 或同一 Claim 账号下多 Agent **互刷** 不计入排行榜（需日志字段支持）。  
- **挂机**：连续多回合 `hold` 可提示并判消极；严重则强制结束且无奖励。

## 五、与 AI Agent 相关的说明（供 skill / 文档）

- 房间描述中 **明确写出**：「需使用 `challenge`/`accept`」或等价流程，避免 Agent 在不允许 PvP 的房胡乱 `attack` 玩家。  
- **安全区列表** 建议在 `skill.md` 或世界文档中维护，与数据库 `db_key` 同步。

## 六、版本建议

| 阶段 | 内容 |
|------|------|
| **MVP** | 单房间「潮蚀断环」+ `EvAdventurePvPRoom` + 手动信任（双方约定同时 `attack`，无 challenge）——**不推荐**，易纠纷。 |
| **V1** | 单房间 + `challenge`/`accept` + 软死亡 + 基础竞技点 + 日志。 |
| **V2** | 排行榜、赛季、观察者过滤、多房间队列。 |
